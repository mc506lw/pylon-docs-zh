## 🔌 插件消息通道详解

Jade2Rebar 使用 Minecraft 的**插件消息通道 (Plugin Messaging Channels)** 系统实现服务端与客户端模组的通信。

### 什么是插件消息通道？

插件消息通道允许服务端插件和客户端模组之间通过自定义通道名称发送和接收数据。这是一种安全且高效的通信方式。

### Jade2Rebar 注册的通道

| 通道名 | 方向 | 用途 | 数据格式 |
| :--- | :--- | :--- | :--- |
| `jade2rebar:server_handshake` | Server → Client | 服务端发起握手请求 | 协议版本、服务器信息 |
| `jade2rebar:client_handshake` | Client → Server | 客户端响应握手 | Jade 版本、模组列表 |
| `jade2rebar:receive_data` | Server → Client | 推送 WAILA 数据 | 方块/实体信息 |
| `jade2rebar:request_block` | Client → Server | 请求方块数据 | 方块坐标 |
| `jade2rebar:request_entity` | Client → Server | 请求数据数据 | 实体 ID |

### 通道注册时机

```kotlin
// 在插件启用时注册所有通道 (Jade2Rebar.kt)
override fun onEnable() {
    // 注册服务端出站通道
    server.messenger.registerOutgoingPluginChannel(
        this, 
        "jade2rebar:server_handshake"
    )
    server.messenger.registerOutgoingPluginChannel(
        this, 
        "jade2rebar:receive_data"
    )
    
    // 注册服务端入站通道
    server.messenger.registerIncomingPluginChannel(
        this,
        "jade2rebar:client_handshake",
        ::handleClientHandshake  // 处理函数
    )
    server.messenger.registerIncomingPluginChannel(
        this,
        "jade2rebar:request_block",
        ::handleBlockRequest
    )
    server.messenger.registerIncomingPluginChannel(
        this,
        "jade2rebar:request_entity",
        ::handleEntityRequest
    )
}
```

## 👥 双轨并行系统

Jade2Rebar 的一个重要设计理念是**双轨并行**：

### 轨道 A：Jade 用户

<Mermaid chart="graph TD
    A[Jade 用户] --> B[WAILA 事件]
    B --> C[拦截转换]
    C --> D[PMC 发送]
    D --> E[增强显示 ✨]" />

### 轨道 B：非 Jade 用户

<Mermaid chart="graph TD
    A[非 Jade 用户] --> B[WAILA 事件]
    B --> C[检测非 Jade]
    C --> D[忽略事件]
    D --> E[原生 WAILA ✓]" />

<Callout type="info">
  这种设计确保了：
  - **兼容性**: 非 Jade 玩家不受任何影响
  - **性能**: 只对 Jade 用户进行额外的数据处理
  - **优雅降级**: 即使 Jade2Rebar 出现问题，Rebar 原生 WAILA 仍然可用
</Callout>

## ⚡ 性能优化策略

Jade2Rebar 采用了多种性能优化策略：

### 1. 延迟检测

不在玩家加入的同一 tick 立即检测，而是等待 40 tick：
- 避免在玩家加载高峰期造成卡顿
- 确保客户端完全就绪

### 2. 智能节流

<Mermaid chart="graph TD
    T1[Tick 1: 触发 → 发送 ✓]
    T2[Tick 2: 触发 → 跳过 ○]
    T3[Tick 3: 触发 → 跳过 ○]
    TN[Tick N: &gt;100ms → 发送 ✓]" />

### 3. 懒惰初始化

只在真正需要时才初始化资源：
- 消息管理器：首次发送消息时加载语言文件
- 转换器：首次转换时初始化

### 4. 内存优化

```kotlin
// 使用轻量级数据结构存储玩家状态
private val jadePlayers = ConcurrentHashMap<UUID, JadePlayerInfo>()

// 而不是存储完整的 Player 对象
data class JadePlayerInfo(
    val uuid: UUID,
    val name: String,
    val detectedAt: Long,
    val clientVersion: String?
)
```

### 5. 异步友好设计

虽然当前版本是同步的，但代码结构支持未来异步化：
- 数据转换逻辑可以轻松移到异步线程
- 数据包发送已经考虑了并发安全

## 🎯 设计哲学总结

Jade2Rebar 的设计遵循以下原则：

1. **透明性**: 对非 Jade 用户完全不可见
2. **可靠性**: 任何组件失败都不会导致崩溃
3. **可配置性**: 几乎所有行为都可调整
4. **可观测性**: 丰富的日志和状态命令
5. **性能优先**: 节流、懒加载、轻量数据结构
6. **向前兼容**: 为未来的异步化和功能扩展预留接口
