## 迈出正确的第一步

如果你一上来就让 AI "帮我写个 XX 附属"，那大概率会翻车——AI 猜不透你的真实意图，写出来的代码往往性能差、不符合预期、甚至根本跑不通。

为了避免这种情况，我们需要**先拆解需求**，把一个大目标拆成多个小任务，每个小任务只做一件事。同时，在动手写代码之前就要想清楚性能最优的方案，避免写到一半发现架构有问题，推倒重来。

在这套教程里，我会带你和 AI 一起做一个经典的例子：**筛子附属（Sieve Addon）**——一个能让玩家把各种方块"筛"成矿物和种子的功能。

### 拆解需求

首先，你得先搞清楚自己到底想要什么。拿出纸笔（还是文件传输助手吧），把你的想法写下来：

1. **核心功能是什么？**
2. **有哪些物品/方块需要自定义？**
3. **交互流程是怎样的？（玩家怎么用）**
4. **有没有概率/数值相关的设定？**

然后，你可以找任何 AI 工具（哪怕是 DeepSeek 的网页版都行），跟它讨论一下你的想法。让它帮你：

- 补充你可能遗漏的细节
- 把需求拆分成具体的物品列表、方块列表、功能点
- 标注出哪些地方可能存在性能问题

举个例子，下面是我给 AI 的需求描述：（直接copy无中生有MOD的设定）

```txt
我想要创建一个 Minecraft Java 服务端插件（基于 Rebar 框架），实现一个筛子系统：

核心玩法：玩家用筛网右键点击筛子，可以把特定方块筛成矿物、种子等产出物。

操作方式：按住右键 3 秒处理一个方块

筛矿概率表（各事件独立，不可叠加）：

泥土 → 小麦种子/草籽(7%)、西瓜/南瓜/甘蔗种子(3%)、萝卜/马铃薯种子(2%)、橡子(2%)、金合欢/云杉/桦木种子(1%) + 石子(必出2个)

沙砾 → 燧石(25%)、煤炭(13%)、青金石(5%)、钻石/绿宝石(1%)、破碎铁/金/铜/锡/铝矿(20%/3%/6%/6%/13%)
```

### 性能优化前置思考

Rebar 本身的设计就非常注重性能，很多常见的性能陷阱框架已经帮我们避开了。但这不代表我们可以完全不管性能——特别是对于高频操作（比如筛子这种按住右键持续触发的功能），如果不提前规划好方案，写出来可能卡到服务器 TPS 暴跌。

在正式开始编码之前，先让 AI 帮你做一次**性能风险评估**：

```txt
基于上面这个筛子插件的设计，可能会出现哪些性能问题？请列出潜在的性能瓶颈，并提供解决方案。
```

基于你设计的筛子插件，以下是从 Minecraft 服务端角度可能出现的性能问题及对应解决方案。这些主要围绕**高频重复操作**、**实体生成**、**方块状态管理**和**玩家交互**展开。

---

**1. 高频随机计算与物品生成（CPU 压力）**

**问题表现**
玩家按住右键每 3 秒完成一次筛选，一次筛选要执行多达十几次独立概率判定（每个产物调用一次 `Random.nextDouble()`）。在服务器同时有多个玩家持续筛矿时，会快速累积计算量，若实现方式粗暴（如使用 BukkitScheduler 每一 tick 都检测），CPU 占用会明显上升。

**解决方案**
- **使用玩家本地计时器而非全局定时任务**
  筛矿过程绑定在玩家交互事件中，不需要每 tick 调度。只在玩家**开始长按**时记录时间戳，在 `PlayerInteractEvent` 或 `PlayerAnimationEvent`（或更推荐用 `PlayerItemHeldEvent` 不适用，最好使用包络检测）中持续比较时间，或使用一个轻量级 `BukkitRunnable` 每 2~3 tick 检查一次即可，而不是每 tick 运行。
- **概率查表预生成**
  对于每个筛矿配方，预先计算**累加分布表**或**权重表**，筛选时只需要一次随机抽取，然后根据表判断哪些产物产出，而不是对每个产物独立调用 `random.nextDouble()`。这样可以大幅减少随机数生成次数。
  例如：将 7% 小麦种子、3% 西瓜种子……转换为区间判定，或使用加权随机选择一次决定所有可能产物（但注意你的需求是"各事件独立，不可叠加"，所以多个产物可能同时产出，无法用一次抽取取代，但仍可通过提前生成一个随机数数组来批量判定，避免多次调用系统随机源）。
  **更优方法**：为每种原料创建一个 `List<Supplier<ItemStack>>` 和一个对应的 `double[]` 概率数组，执行判定时遍历一次，使用同一个 `ThreadLocalRandom` 实例，开销极低。高并发下注意使用 `ThreadLocalRandom` 而非共享 `Random` 实例。
- **合并产物生成**
  筛选完成后将产生的所有物品合并到一个 `List<Item>`，一次性生成掉落物实体（或直接尝试放入玩家背包），避免多次生成实体时的重复物理计算。
...
**总结优化优先级**

| 性能影响 | 推荐优先实施 |
|--------|------------|
| 掉落物实体过多 | 产物直接进背包 |
| 高频随机判定 | 概率预计算、使用 ThreadLocalRandom |
| 长按检测空转 | 事件驱动 + 定时单次任务 |
| 方块更新开销 | 用实体展示内容而非方块状态 |
| 内存泄漏 | 完善的方块破坏/玩家退出监听清理 |
| 配置重复读取 | 配置缓存 + 热重载 |

按上述方案实现后，即使服务器同时有几十名玩家长期连续筛矿，对 TPS 的影响也可控制在可忽略范围（通常 1% CPU 增量）。若你计划开始编码，我可以进一步为你拆分具体的类职责和事件监听结构。
</CollapsibleSection>

<Callout type="note">

</Callout>
