## 工作原理

Cargo 通过将 `RebarCargoBlock` 使用 `CargoDuct` 连接在一起来工作。Cargo 与流体的不同之处在于，物流管道不能有 junction：每段物流管道只连接两个端点。

## RebarCargoBlock

`RebarCargoBlock` 代表一个物流端点。当一个方块实现 `RebarCargoBlock` 时，它可以在面上添加物流组（参见 `RebarLogisticBlock`），指定面类型（INPUT 或 OUTPUT）。当日志组与某个面和一个类型关联时，物流管道可以连接到该面。

## 物品路由

物流管道中的一个主要问题是连接的任何部分都可能随时卸载，考虑到连接可以几乎是无限长的。这在路由方面实际上非常困难。我们可以只是在内存中存储哪些端点与哪些节点连接在一起，但每当网络发生任何更新时，都必须更新这个存储。

我能（Idra）找到的最简单的解决方法是使用缓存。每当我们想知道输出连接到哪个输入时，我们都会查询缓存。如果缓存已经计算了路线，则返回该路线。否则，我们从输出开始计算路线，跟随所有物流管道直到到达输入。记录所有物流管道。每当路线上的任何块被修改（破坏或卸载）时，整条路线将被删除并且必须重新计算。

这意味着如果物流管道网络的中间部分被卸载，路线将失效，物品将不再沿着该网络路由。

## Cargo  ticking

每个 `RebarCargoBlock` 都会定期 tick。一个 cargo 块 tick 如下所示：

1. 遍历 cargo 块上的每个输出面（及其对应的物流组）
2. 使用上述缓存查找对应的输入块+面（如果存在）（及对应的物流组）
3. 遍历源物流组中的每个源槽位
4. 遍历目标物流组中的每个目标槽位
5. 检查是否可以将物品从源槽位转移到目标槽位（例如，如果源包含一些西瓜，目标什么都不包含或也包含西瓜且少于 64 个）
6. 为您的电脑即将爆炸而担忧
7. 一旦我们成功转移任何物品或用完要遍历的源或目标槽位，我们就完成了

## CargoDuct

物流管道跟踪它们连接的面并管理相关的显示。这基本上是它们的唯一功能。每当物流块或物流管道与它们相邻时，它们会检查是否可以连接到该管道/块。如果可以，将其添加到连接面的列表中，刷新物品显示，然后也更新另一个块并刷新它的物品显示。

物品显示刷新主要涉及查看所有连接的面，并确保每个面都有一个相关的显示。（它还会删除任何不应再存在的显示）。

此外，当物流管道或块连接/断开时会触发事件。

## Cargo 块 hack

Cargo 块**不**跟踪连接的面。"你他*的在说什么？"你可能会说。"Cargo 块不能直接连接在一起，而不需要在中间使用物流管道？！”是的，但这是一个 hack。基本上，当创建物品路由逻辑时，它已经适用于直接相邻的 cargo 块，所以需要做的只是让显示正确更新。为此，创建了 `updateDirectlyConnectedFaces` 函数，它基本上只是检查所有相邻的 cargo 块以确保所有显示都是正确的。连接两个 cargo 块在一起的显示实际上由其中一个特定的 cargo 块拥有（不是所有 cargo 块，如物流管道那样）。这只是一个巨大的 hack。

## Z-fighting

必须小心避免 Z-fighting。连接管道在一起的显示实际上都有不同的宽度，否则它们会重叠和 Z-fighting。这是通过使用 3 种不同的宽度并确保当我们在一或两个角上连接两个管道时，在角上管道现有显示（如果存在）之间生成的新显示不是它已经使用的任何宽度。希望这讲得通
