## 流体点是什么？

流体点是您必须点击来放置管道的红/绿/灰方块。有 3 种类型的流体点：`INPUT`（绿色）、`OUTPUT`（红色）和 `CONNECTOR`（灰色）。输入和输出点附加到方块上以允许添加/移除流体，而 connector 仅用于连接其他点。

要为您的机器创建流体连接点，您可以调用 `FluidPointInteraction.make(...)`。

## 流体方块

### PylonFluidBlock

任何具有流体输入/输出的方块都必须实现 `PylonFluidBlock`。此接口允许您从输入点请求流体、向输出点供应流体，并指定如何从您的方块添加/移除流体。困惑？请参阅"ticking"部分以了解其工作原理。

有关用法信息，请参阅 PylonFluidBlock Javadocs。

### PylonFluidBufferBlock

通常，流体机器在内部流体中存储流体。例如，压榨机有一个默认大小为 1000mB 的内部缓冲区，用于储存植物油。这是如此常见的情况，以至于我们创建了一个新接口来处理它：PylonFluidBufferBlock。此接口允许您的块轻松管理流体缓冲区。

有关用法信息，请参阅 PylonFluidBufferBlock Javadocs。

### PylonFluidTank

另一种常见模式是"流体 tanks"，它一次只能存储一种流体，但可以存储多种类型的流体。PylonFluidTank 实现了这种模式。

有关用法信息，请参阅 PylonFluidTank Javadocs。

## 物理与虚拟流体点

有物理流体点和虚拟流体点。"物理"流体点（用于显示框并允许您与之交互的显示和交互）包含一个"虚拟"流体点。这种区分可能看起来很令人困惑，但有助于保持底层逻辑简单。

物理流体点主要是虚拟流体点的包装器，它们提供显示它们的逻辑和处理玩家与它们的交互 - 它们不是太重要，您几乎永远不需要考虑它们，所以当我们说"流体点"时，我们指的是虚拟流体点。

虚拟流体点有几个属性，包括流体点的 UUID（用于容易引用流体点）、流体点的类型、连接的流体点和流体点的方块。如果流体点是输入或输出，其方块必须是 PylonFluidBlock。

## 分段

流体在"分段"的帮助下进行管理。分段只是一组连接的流体点。分段封装**所有**连接的流体点。例如，这里（破折号表示管道），A、B 和 C 都在一个分段中，D 和 E 在另一个分段中，F 在又一个分段中：
```
A---B-----C  D--E    F
```

如果我们移除 B 和 C 之间的管道，那么 A 和 B 将形成一个新的分段，C 也将形成一个新的分段。旧分段将被删除。相反，如果我们连接 C 和 D，那么 A/B/C 和 D/E 分段将合并成一个新分段。

## Ticking

分段很重要，因为它们是流体 ticking 工作原理的基础。每个分段每几个游戏刻 tick 一次（取决于您的服务器配置）。当流体分段被 tick 时，会发生以下情况：

1. 我们调用分段中每个块的 `getRequestedFluids` 并列出正请求什么流体以及每种流体请求了多少。
2. 我们遍历每个供应的流体。
3. 如果流体不被分段允许，我们跳过它。
4. 如果没有机器请求流体（即，对于每个机器 `fluidAmountRequested` 返回零），我们跳过它。
5. 当/如果我们找到至少有一个机器请求的流体时，我们使用轮询算法尽可能从每个机器获取流体，具体取决于可以通过管道传输多少流体以及请求了多少。
6. 我们进行另一轮轮询，但这次是针对我们发送流体的机器，以便在它们之间均匀分配流体。

## 流体流速和允许的流体

我们通常希望限制管道传输流体的速度。正确执行这一点很困难且计算成本高，所以我们只是"撒谎"并改为**每个分段**限制流体流速。我们也可能希望限制可以通过管道的流体类型（例如，热流体不允许在木管道中）。这又是按每个分段进行的。

部分原因就是为什么您不能将不同类型的管道连接在一起：我们使用哪个管道的流体流速作为分段的？

当管道分段合并时， arbitrarily 选择一个分段并保留其流体流速和允许的流体。当管道分段分裂时，旧分段的流体流速和允许的流体被保留到新分段。
